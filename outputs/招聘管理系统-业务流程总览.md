# 招聘管理系统-业务流程总览

> 用途：这是从 `招聘管理系统_功能详细清单_v3.0.docx` 和 `docs/prd/` 提炼出的最小可执行业务流程图。后续开发与验证时，优先检查业务链条是否能跑通，而不是只看单个按钮或单个模块是否“有页面”。

## 1. 权威来源

- `招聘管理系统_功能详细清单_v3.0.docx`
- `PRD.md`
- `docs/prd/`

如果这些文档之间有冲突，以产品最终 DOCX 为准。

## 2. 核心业务主线

```mermaid
flowchart LR
  A[简历导入 / 人工建档] --> B[候选人数据池]
  B --> C[候选人筛选与锁定]
  C --> D[推荐到岗位]
  D --> E[面试跟踪]
  E --> F[录用 / 入职]
  F --> G[交付记录]
  G --> H[质保跟踪]
  H --> I[质保结束 / 重新激活]
```

## 3. 业务链条拆解

### 3.1 入口：简历与候选人

1. 简历从外部来源导入，或由运营人员人工建档。
2. 系统生成候选人基础档案，进入候选人数据池。
3. 候选人可以被搜索、筛选、预览、打标签、评价和导出。
4. 候选人可被锁定，避免重复推进；条件满足后可释放或失效。

### 3.2 需求进入：客户、项目、岗位

1. 客户公司进入系统。
2. 客户下创建项目。
3. 项目下拆分岗位。
4. 岗位承接后续推荐、面试、录用和交付信息。

### 3.3 推进：推荐、面试、录用、交付

1. 候选人从数据池进入岗位推荐。
2. 推荐后进入面试跟踪。
3. 面试结果会沉淀评价记录。
4. 候选人录用后形成交付记录。
5. 交付记录进入质保期管理。

### 3.4 收口：质保与复用

1. 入职后按质保期持续跟踪。
2. 质保到期前后可触发提醒、重新激活或关闭。
3. 所有关键动作应保留日志、通知和状态留痕。

## 4. 验证时的检查顺序

每次验证不要只看单页结果，按下面顺序走：

1. 入口是否真实写库。
2. 列表是否能刷新看到新数据。
3. 关联页是否能接到同一条业务对象。
4. 状态流转后，下一步动作是否还可继续。
5. 日志、通知、统计是否同步反映。
6. 流程末端是否能进入质保或关闭态。

## 5. 常见断点

- 只有弹窗，没有真实写库
- 写了库，但关联页面没有刷新
- 详情页能看，后续推荐或面试接不起来
- 推荐有了，但交付、质保没有后续状态
- 只验证模块存在，没验证整条链路能走通

## 6. 当前执行原则

- 开发和验收都优先检查流程闭环。
- 只要业务对象涉及流转，就必须确认下一步还能接上。
- 如果某个功能看起来像“点一下直接完成”，先回到 PRD 检查它是否应该弹窗确认、补充字段或走状态流转。
