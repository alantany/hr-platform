# AGENTS.md

## 1. 项目定位

- 这是一个静态的猎头招聘交付系统原型，目标是验证页面结构、交付流程和信息组织方式。
- 当前工作重点是维护原型一致性，而不是引入额外框架或重构成复杂工程。
- 所有修改都应优先服务于招聘管理场景中的客户、项目、岗位、候选人、评价、质保和权限管理。

## 2. 默认技术栈

- 前端：`Vue 3 + TypeScript`
- 后端：`Python + FastAPI`
- 数据库：`PostgreSQL`
- ORM：`SQLAlchemy 2.0`
- 迁移：`Alembic`

说明：
- 这是后续新开发默认沿用的技术栈。
- 如果用户明确要求不同方案，以用户要求为准。
- 在原型阶段，优先保持前后端边界清晰，不要把业务逻辑塞进页面文件。

## 3. 仓库结构

- 仓库根目录：放共享脚本、全局样式、运行入口和项目级说明。
- `src/pages/`：放所有 HTML 页面，页面文件不要再散落到仓库根目录。
- `app.js`：负责共享壳层、导航、页面标题、面包屑和页内元信息。
- `styles.css`：负责全局视觉系统、布局和响应式样式。
- `outputs/`：放原型理解、业务框架、阶段性分析等文档，不是页面资源目录。
- `task_plan.md`、`findings.md`、`progress.md`：是当前任务的工作记录文件，优先保留并持续更新。

## 4. 参考文档

优先参考以下文件理解业务和当前状态：

- `outputs/猎头招聘交付系统-原型理解框架.md`
- `task_plan.md`
- `findings.md`
- `progress.md`

使用方式：
- `outputs/猎头招聘交付系统-原型理解框架.md` 用来理解系统边界、业务对象和页面定位。
- `task_plan.md` 用来理解当前任务的目标、范围和阶段。
- `findings.md` 用来查看已经确认的事实、约束和调研结论。
- `progress.md` 用来查看最近做过什么、测试到哪一步、有没有未完成事项。

## 5. 工作原则

- 优先做小而聚焦的修改，尽量保持现有原型结构。
- 除非用户明确要求重设计，否则保留现有视觉语言和页面组织方式。
- 文件名、页面路由、导航名称和页面标题要保持一致。
- 如果新增页面，先沿用现有命名和导航风格，再同步更新 `app.js`。
- 除非用户明确要求，不要引入新的框架依赖或额外复杂度。
- 修改业务文案时，优先和 `outputs/猎头招聘交付系统-原型理解框架.md` 对齐。

## 6. 页面入口

- `src/pages/index.html` 负责跳转到仪表盘。
- `src/pages/dashboard.html` 是主入口页面。
- `app.js` 是页面壳层和导航的统一入口，改导航、标题、面包屑时优先检查这里。
- `styles.css` 是所有页面的共同样式基座，改布局和视觉时优先看这里。

## 7. 验证方式

- 对静态 HTML/CSS/JS 的修改，优先在浏览器里打开对应页面验证。
- 如果涉及路由或页面搬迁，要确认链接仍然能正确跳转。
- 如果修改了布局或样式，要同时检查桌面宽度和窄屏表现。
- 如果修改了 `app.js` 的共享壳层，至少要核对首页、仪表盘和一个普通功能页是否都能正常渲染。
- 如果修改的是文案、注释或文档，优先做一致性检查，不必额外引入复杂测试。
- 如果工作包含多个阶段，要把关键结论写回 `findings.md`，把当前进度写回 `progress.md`。
- 任何会写入数据库、导出文件、临时截图或测试附件的验证/自动化测试，都必须在测试结束时同步清理现场；优先使用 `tests/conftest.py` 这类公共收尾机制，单个测试若创建了独立脏数据，也必须在 `finally` 中回收，禁止把测试数据留在仓库或数据库里。

## 8. 任务记录

- 开始多步骤工作前，先确认 `task_plan.md`、`findings.md`、`progress.md` 是否已经存在。
- 做出结构性判断后，优先把结论落到 `findings.md`，不要只留在对话里。
- 每完成一个阶段，更新 `progress.md`，必要时同步调整 `task_plan.md`。
- 只要完成了一个实际开发任务，不论大小，都应在结束前补写或更新 `progress.md`；如果任务有分阶段推进，再同步更新 `task_plan.md`。
- 如果发现与当前目标相关的新约束，要先记录，再继续执行。
- 仓库已配置本地 pre-commit hook；如果修改了代码或文档却没有同步 `progress.md`，提交前应先运行收尾脚本并补齐记录。
- 新增或修改测试时，要同步检查测试是否会写入表数据、文件或截图；如果会写，必须同时更新清理逻辑和对应记录，确保测试结束后数据库保持干净。

## 9. Git 和安全

- 不要覆盖用户已有修改，除非用户明确要求。
- 优先使用非破坏性的 git 命令。
- 除非用户要求合并流程，否则提交和推送分开处理。
- 不要为了“看起来整洁”而删除用户未确认的工作文件。
- 每次完成一次实际代码或文档修改后，默认执行 `git commit` 并 `git push`，除非用户明确要求不要提交或不要推送。
